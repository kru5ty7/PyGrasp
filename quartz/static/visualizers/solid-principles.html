SOLID Principles — Click a card to expand

S
Single Responsibility
One class, one reason to change
O
Open / Closed
Extend without modifying
L
Liskov Substitution
Subclass can replace parent
I
Interface Segregation
No forced unused methods
D
Dependency Inversion
Depend on abstractions
S — Single Responsibility Principle
❌ BAD — Fat class
UserManager
- user_data
- email_config
- db_conn
+ save_user()
+ send_email()
+ hash_password()
+ log_activity()
4 responsibilities — every concern coupled together
✓ GOOD — Split classes
User
+ hash_password()
EmailService
+ send_email()
UserRepository
+ save()
Each class changes for exactly one reason
O — Open/Closed Principle
❌ BAD — if/elif chain
def area(shape): if shape == 'circle': ... elif shape == 'rect': ... # new shape = edit this!
Every new shape requires modifying existing code
✓ GOOD — Polymorphism
Shape (abstract)
+ area(): float
Circle, Rectangle, Triangle all extend Shape
# AreaCalculator never changes def total(shapes): return sum(s.area() for s in shapes)
New shape = new subclass only
L — Liskov Substitution Principle
❌ BAD — Broken subtype
Bird
+ fly()
extends ↓
Penguin
+ fly()
raise NotImplementedError!
Subclass breaks parent contract
✓ GOOD — Correct hierarchy
Bird
+ name
FlyingBird extends Bird  |  NonFlyingBird extends Bird
FlyingBird
+ fly()
Penguin extends NonFlyingBird — no fly() needed
I — Interface Segregation Principle
❌ BAD — Fat interface
IWorker
+ work()
+ eat()
+ sleep()
+ charge_battery()
Robot must implement eat() and sleep() — makes no sense
✓ GOOD — Segregated
IWorkable
+ work()
IHuman
+ eat()
+ sleep()
IChargeable
+ charge()
Each class implements only what it needs
D — Dependency Inversion Principle
❌ BAD — Concrete dependency
OrderService
+ place_order()
— depends on concrete ↓
MySQLDatabase
+ save()
Swap DB = modify OrderService
✓ GOOD — Abstraction layer
OrderService
+ place_order()
— depends on abstraction ↓
DatabaseInterface
+ save()
MySQL / Postgres both implement it
Swap DB = just swap the implementation
