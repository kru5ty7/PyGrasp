# Vector Databases

> Vector databases store embeddings and perform fast approximate nearest neighbor (ANN) search — `pgvector` (PostgreSQL extension), Pinecone, Chroma, and Weaviate are common choices; they index vectors for sub-second similarity search over millions of embeddings.

---

## Quick Reference

**Core idea:**
- Vector DB: stores `(id, vector, metadata)` and supports similarity search (`find k nearest vectors to query_vector`)
- **ANN (Approximate Nearest Neighbor)**: trades perfect accuracy for speed — finds the closest matches, not guaranteed the absolute closest; acceptable for RAG
- **HNSW (Hierarchical Navigable Small Worlds)**: the most common ANN index algorithm; fast queries, moderate memory
- `pgvector`: PostgreSQL extension — adds `vector` type and `<=>` (cosine), `<->` (L2), `<#>` (inner product) operators
- Metadata filtering: filter by metadata before or after vector search to narrow results

**Tricky points:**
- ANN is approximate — it may miss the true nearest neighbor; HNSW's `ef_search` parameter trades accuracy for speed
- Dimensionality: OpenAI `text-embedding-3-large` = 3072 dims; `text-embedding-3-small` = 1536 dims; Claude embeddings (Voyage) = 1024 dims; higher dims = slower
- Index build time: HNSW requires full index rebuild for large new batches — use `IVFFlat` for frequent inserts; HNSW for stable datasets
- Metadata filtering + vector search: pre-filter (filter first, then search) vs. post-filter (search, then filter) — pre-filter reduces the search space but may miss relevant results if the filter is too restrictive
- pgvector is the simplest option for projects already using PostgreSQL — no separate service to manage

---

## What It Is

A vector database is optimized for one operation: given a query vector, find the k most similar vectors in the store. Relational databases can do this (`ORDER BY vector <=> query_vector LIMIT 5`) but it requires scanning every row — slow for millions of rows. Vector databases build specialized indexes (HNSW, IVF) that enable approximate search in milliseconds regardless of scale.

For RAG: embed your documents → store in vector DB → at query time, embed the question → search vector DB → retrieve top-k documents → include in LLM prompt.

## How It Actually Works

`pgvector` with SQLAlchemy:
```python
from sqlalchemy import Column, Integer, String
from pgvector.sqlalchemy import Vector

class Document(Base):
    __tablename__ = "documents"
    id = Column(Integer, primary_key=True)
    content = Column(String)
    embedding = Column(Vector(1536))  # dimension must match embedding model

# Create HNSW index:
# CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);

def search_similar(query_embedding: list[float], k: int = 5) -> list[Document]:
    return session.scalars(
        select(Document)
        .order_by(Document.embedding.cosine_distance(query_embedding))
        .limit(k)
    ).all()
```

Chroma (in-memory, good for prototyping):
```python
import chromadb
from chromadb.utils import embedding_functions

client = chromadb.Client()  # or chromadb.PersistentClient(path="./chroma_db")
ef = embedding_functions.SentenceTransformerEmbeddingFunction(model_name="all-MiniLM-L6-v2")

collection = client.create_collection(
    name="documents",
    embedding_function=ef,
    metadata={"hnsw:space": "cosine"}
)

# Add documents (Chroma embeds automatically):
collection.add(
    ids=["doc1", "doc2"],
    documents=["FastAPI is great", "SQLAlchemy helps with databases"],
    metadatas=[{"source": "tutorial"}, {"source": "docs"}]
)

# Search:
results = collection.query(
    query_texts=["how to use web frameworks"],
    n_results=2,
)
```

## Common Misconceptions

Misconception 1: "pgvector is too slow for production."
Reality: pgvector with an HNSW index handles millions of vectors with sub-100ms query times on typical hardware. For most applications (under 10M vectors), pgvector is sufficient. Dedicated vector DBs (Pinecone, Weaviate) add operational complexity that only pays off at very large scale.

Misconception 2: "Vector databases replace traditional databases."
Reality: Vector databases excel at similarity search. They're not replacements for relational databases — most production systems use both: relational DB for structured data + vector index (either pgvector or separate service) for semantic search.

## Why It Matters in Practice

Choosing a vector database:
```
pgvector:       Already using PostgreSQL; simple ops; up to ~10M vectors
Chroma:         Local dev, prototyping, open source
Pinecone:       Managed, large scale, no ops
Weaviate:       Open source, self-hosted, multimodal
Qdrant:         Open source, performance-focused, Rust backend
```

For most FastAPI + RAG applications: start with `pgvector` — no additional service, battle-tested PostgreSQL, full SQL support for metadata filtering.

Answer frame: Vector DB stores embeddings + metadata; optimized for ANN search (find k nearest vectors). HNSW index for fast approximate search. Key trade-off: ANN is approximate — faster than exact search, nearly as accurate. `pgvector` for existing PostgreSQL deployments. Metadata filtering narrows results.
