# Tokenization

> Tokenization is the first stage of Python's compilation pipeline — the lexer reads raw source characters and produces a stream of typed tokens (NAME, NUMBER, STRING, OP, NEWLINE, INDENT, DEDENT) that the parser can work with.

---

## Quick Reference

**Core idea:**
- The **lexer** (tokenizer) converts raw source text characters into a sequence of **tokens** — typed, structured units like `NAME('def')`, `OP(':')`, `NEWLINE`, `INDENT`
- Python's `tokenize` module exposes the lexer — `tokenize.tokenize(source)` yields `TokenInfo` named tuples with `(type, string, start, end, line)`
- Token types: `NAME` (identifiers and keywords), `NUMBER`, `STRING`, `OP` (operators and punctuation), `NEWLINE`, `INDENT`, `DEDENT`, `COMMENT`, `ENCODING`
- **`INDENT` and `DEDENT`** are synthetic tokens generated by the tokenizer to represent Python's significant whitespace — the parser never sees raw spaces
- **Comments and whitespace** (except for INDENT/DEDENT) are stripped by the tokenizer — the parser receives a clean token stream

**Tricky points:**
- Python's tokenizer handles **multiline strings** as single `STRING` tokens — the entire triple-quoted block is one token
- `INDENT` and `DEDENT` are generated by tracking an **indentation stack** — the tokenizer pushes each new indentation level and pops when indentation decreases, emitting `DEDENT` tokens for each popped level
- **Encoding detection** happens at tokenization time — the tokenizer reads the encoding cookie (`# -*- coding: utf-8 -*-`) and decodes the file accordingly
- Tokenization errors (e.g., unclosed string literal) raise `TokenError` before any parsing occurs
- The tokenizer is intentionally **context-free** — `def` is always tokenized as a `NAME` token, not as a keyword; keyword disambiguation happens in the parser

---

## What It Is

Think of reading a sentence and breaking it into words and punctuation before trying to understand its grammar. Before a grammar checker can analyze "The quick brown fox jumps over the lazy dog," it needs to recognize the individual words and punctuation as distinct units. It does not try to parse grammar from a stream of individual letters. Tokenization is that breaking-into-words step for Python source code. The raw file is a sequence of characters; the tokenizer produces a sequence of meaningful units — `def`, `my_function`, `(`, `x`, `:`, `int`, `)`, `->`, `None`, `:` — before any grammar analysis.

Tokenization separates the concern of "what are the basic units of syntax" from "how do those units form valid structures." The tokenizer knows about Python identifiers, string delimiters, number formats, and indentation rules. It does not know about function definitions, class hierarchies, or expression precedence — those are the parser's concerns. This separation keeps each stage simpler and allows the parser to work with a clean token stream rather than raw characters.

Python's tokenizer has one particularly important responsibility that most tokenizers lack: generating `INDENT` and `DEDENT` tokens to encode Python's significant whitespace. Most languages use explicit braces or keywords to delimit blocks; Python uses indentation. The tokenizer converts raw whitespace at the start of lines into virtual `INDENT` (more indented than previous line) and `DEDENT` (less indented) tokens. The parser then treats these tokens exactly like opening and closing braces — they delimit block boundaries without the parser needing to understand whitespace at all.

## How It Actually Works

CPython's tokenizer is implemented in `Python/tokenize.c` and `Lib/tokenize.py` (the pure-Python version for tooling). The tokenizer reads the source file character by character, maintaining a state machine that tracks the current token being assembled. When a token boundary is detected (a space after a name, an operator character, a closing quote), the completed token is emitted as a `TokenInfo` object.

Indentation tracking is the most complex part. The tokenizer maintains an indentation stack (a list of indentation levels, measured in space-equivalent columns). At the start of each logical line, it measures the current indentation. If greater than the stack top, it pushes the new level and emits an `INDENT` token. If equal, nothing is emitted. If less, it pops levels from the stack (emitting a `DEDENT` for each popped level) until the stack top matches the current indentation. If the current indentation matches no level in the stack, an `IndentationError` is raised.

The `tokenize` module in the standard library provides Python-level access to this process. `import tokenize, io; tokens = tokenize.tokenize(io.BytesIO(source).readline)` yields a stream of `TokenInfo` named tuples. Inspecting this stream for a short Python function reveals the exact token sequence the parser will receive: `ENCODING`, `NAME('def')`, `NAME('greet')`, `OP('(')`, `NAME('name')`, `OP(')')`, `OP(':')`, `NEWLINE`, `INDENT`, `NAME('return')`, `STRING('"Hello "')`, `OP('+')`, `NAME('name')`, `NEWLINE`, `DEDENT`, `ENDMARKER`.

## How It Connects

Tokenization is the first stage in the pipeline that leads from source text to bytecode. The token stream produced by the tokenizer feeds directly into the parser, which builds an Abstract Syntax Tree (AST) from the token sequence. Understanding tokenization explains why `IndentationError` and `SyntaxError` can be raised before any code executes.

## Common Misconceptions

Misconception 1: "Python keywords are special tokens with their own token type."
Reality: In Python's tokenizer, `def`, `class`, `if`, `for`, `return`, and all other keywords are tokenized as `NAME` tokens — the same type as user-defined identifiers. The string value distinguishes them: `NAME('def')` vs `NAME('my_var')`. The parser, not the tokenizer, treats keywords differently based on their string value. This design keeps the tokenizer simple and allows keywords to be used as attribute names (which is legal: `obj.class` is invalid due to the parser's treatment of `class`, but the tokenizer produces a valid `NAME` token for it).

Misconception 2: "Comments are preserved in the token stream for the parser."
Reality: Comment tokens (`COMMENT`) are included in the token stream produced by `tokenize.tokenize()`, but CPython's actual compilation pipeline strips comments before the token stream reaches the parser. The AST has no representation for comments — they are discarded during tokenization from the compiler's perspective. Tools that need to preserve comments (reformatters like Black, documentation generators) use the `tokenize` module directly rather than going through the AST.

## Why It Matters in Practice

The `tokenize` module is the foundation for Python code transformation tools. Formatters (Black, autopep8), import sorters (isort), and syntax highlighters all operate on the token stream rather than the AST, because token positions map directly to character positions in the source file — allowing round-trip transformation without reformatting the entire file. When you want to rename all occurrences of a string literal or rewrite import statements, working at the token level preserves all comments and formatting outside the targeted tokens.

`SyntaxError` and `IndentationError` both originate in the tokenizer or parser. An `IndentationError` is raised by the tokenizer when the indentation stack cannot be reconciled. A `SyntaxError` for an unclosed string literal is raised by the tokenizer. A `SyntaxError` for a malformed expression (like `x = (1 +`) is raised by the parser after the tokenizer has already produced valid tokens. Knowing which stage raises which error helps diagnose problems: if you see an `IndentationError`, look at the preceding block structure; if you see a general `SyntaxError`, look at the expression structure.

## Interview Angle

Common question forms:
- "What does the Python tokenizer do?"
- "How does Python handle significant indentation?"
- "What is the difference between tokenization and parsing?"

Answer frame: The tokenizer converts raw source characters to a typed token stream: NAME, NUMBER, STRING, OP, NEWLINE, INDENT, DEDENT. It handles Python's significant whitespace by maintaining an indentation stack and generating synthetic INDENT/DEDENT tokens. The parser receives the clean token stream and builds the AST — it never sees raw spaces. Keywords are tokenized as NAME tokens; the parser distinguishes them. Comments are stripped at tokenization. `tokenize` module provides Python-level access to the token stream.
