# Named Tuples

> A named tuple is a tuple where each position has a name  -  giving you the clarity of a class with the memory efficiency and immutability of a tuple.

---

## Quick Reference

**Core idea:**
- `collections.namedtuple('Point', ['x', 'y'])` generates and returns a new class (a tuple subclass)
- Memory layout: identical to plain tuple  -  contiguous C array of pointers, no `__dict__`, no per-instance overhead
- Field access by name uses class-level descriptors that call `tuple.__getitem__(self, index)`
- Three key helper methods: `_asdict()` -> `dict`, `_replace(**kwargs)` -> new instance, `_fields` -> field name tuple
- `typing.NamedTuple` syntax: class-level annotations, supports default values, preferred for new code

**Tricky points:**
- `namedtuple` is a factory that returns a class  -  `Point = namedtuple(...)` not `p = namedtuple(...)`
- Named tuples are fully tuple-compatible: `isinstance(p, tuple)` is `True`, unpacking works, indexing works
- Cannot add new fields or change existing values  -  same immutability as tuple
- `_replace()` returns a new instance, it does not mutate in place  -  immutability is preserved
- Default values in `collections.namedtuple` require the `defaults` parameter (Python 3.6.1+): `namedtuple('P', ['x', 'y'], defaults=[0])`

## What It Is

Imagine receiving a parcel with no labels  -  you know slot 0 is the tracking number and slot 1 is the destination, but only because you memorized the protocol. Now imagine the same parcel with labeled compartments: "tracking_number", "destination". You access the same data with the same speed, but the code reads like English instead of like an array index. The parcel itself weighs exactly the same  -  the labels are on the box design, not stamped on every individual item inside.

That is the named tuple. It is a tuple that has been given a schema  -  a fixed set of named positions. The names are stored once on the class, not once per instance. Every instance of `Point(x=1, y=2)` is an honest tuple: `isinstance(p, tuple)` returns `True`, `x, y = p` works, `p[0]` returns `1`. But `p.x` also returns `1`, and the name `x` documents what that position means.

Named tuples are the right type for value objects that are small, fixed-shape, immutable, and tuple-compatible. A geographic coordinate pair, a database row, a function return type with two or three fields  -  these are natural fits. They occupy less memory than a class with `__init__` because there is no per-instance `__dict__`. They are faster to construct than dataclasses for simple cases because the generated `__new__` is lean. And they serialize to tuples naturally, which matters in contexts like `csv.writer` or `struct.pack`.

## How It Actually Works

`collections.namedtuple` is implemented as a function that builds a string of Python class source code, compiles it, and `exec`s it into a temporary namespace. The technique has evolved across Python versions  -  in recent CPython it uses the `_nt_itemgetters` helper and class construction rather than raw string exec  -  but the result is a genuine class object living in the interpreter's memory, indistinguishable from a hand-written class.

The generated class inherits from `tuple`. Field access is provided by `property` descriptors at the class level:

```python
# Conceptually what namedtuple generates for Point.x:
class Point(tuple):
    __slots__ = ()
    x = property(lambda self: tuple.__getitem__(self, 0))
    y = property(lambda self: tuple.__getitem__(self, 1))
    # ... plus __new__, __repr__, _asdict, _replace, _fields
```

`__slots__ = ()` prevents the creation of a per-instance `__dict__`, which is why named tuples have the same memory footprint as plain tuples. The `property` objects live on the class (one allocation per class, not per instance), and each field access calls `tuple.__getitem__`  -  no dictionary lookup, direct slot access.

`typing.NamedTuple` provides a different syntax for the same result:

```python
from typing import NamedTuple

class Point(NamedTuple):
    x: float
    y: float = 0.0   # default value supported here
```

This syntax is processed by the metaclass machinery during class creation. It calls `collections.namedtuple` internally, but the class-body syntax makes type annotations available to type checkers and keeps field definitions co-located with the class name. The `defaults` support is cleaner here than in the `namedtuple()` call form.

## Common Misconceptions

Misconception 1: "`namedtuple('Point', ['x', 'y'])` gives you a Point instance."
Reality: It gives you a Point class. You must then call `Point(1, 2)` to create an instance. The variable you bind it to becomes the class name by convention  -  `Point = namedtuple('Point', ['x', 'y'])`.

Misconception 2: "Named tuples are just fancy dicts."
Reality: Named tuples are tuple subclasses  -  they are ordered, indexed, iterable, hashable (when elements are hashable), and memory-efficient. They do not have the O(1) key lookup of a dict. Field access by name goes through a descriptor that calls `tuple.__getitem__`  -  it is O(1) but via a descriptor, not a hash table.

Misconception 3: "`_asdict()` returns an `OrderedDict`."
Reality: In Python 3.8+, `_asdict()` returns a regular `dict` (insertion-ordered since 3.7). In Python 3.7 and below, it returned an `OrderedDict`. Code relying on `_asdict()` returning an `OrderedDict` for its ordering type should be updated.

## Why It Matters in Practice

Named tuples are the right first choice when returning multiple values from a function by name without reaching for a full class. A function that returns `(latitude, longitude, altitude)` is hard to use  -  callers must remember what position 0 means. A function that returns `Location(lat=..., lon=..., alt=...)` is self-documenting, equally fast, and backward-compatible with any code that unpacks it positionally.

They also matter for performance-sensitive code that processes many small records. A million `namedtuple` instances use roughly the same memory as a million plain tuples. A million `dataclass` instances each carry a `__dict__`, adding significant overhead unless `slots=True` is used. For read-only record types at scale, the named tuple is the lightest well-named container available.

## Interview Angle

Common question forms:
- "What is the difference between a named tuple and a dataclass?"
- "How does named tuple field access work internally?"
- "When would you use `typing.NamedTuple` over `collections.namedtuple`?"

Answer frame:
Named tuples are immutable tuple subclasses  -  no `__dict__`, same memory as tuple, hashable if elements are. Dataclasses are regular classes with `__dict__` (unless `slots=True`), support mutation, and can have default factories. Field access uses class-level `property` descriptors that call `tuple.__getitem__`. Prefer `typing.NamedTuple` syntax for type annotations and default value support.
